# 음성 인식 정확도 향상 작업 계획서 (Revised)

## 0) 문서 목적
본 문서는 음성 입력(STT) 분석 정확도를 체계적으로 개선하기 위한 통합 계획서이다. 단순한 모델 성능 향상을 넘어, **실제 앱 사용성을 극대화**하기 위한 앱 내 로직 개선, 데이터 수집, 그리고 지속적인 개선을 위한 **Flywheel(선순환) 구조** 설계를 포함한다.

## 1) 목표
- **사용자 체감 정확도 극대화**: 단순 CER(문자 오류율) 감소보다, **핵심 정보(금액, 날짜, 장소)**의 인식률 보장
- **비용 효율적 Fallback 구조**: Local STT로 충분한 경우를 판별하여 고비용/고지연 Whisper 호출 최소화
- **자가 발전 시스템 구축**: 사용자의 수정 행동이 곧 학습 데이터가 되는 파이프라인 마련

## 2) 현재 갭 분석 (Gap Analysis)
- **로그 맥락 부족**: 오디오와 인식 결과가 존재하나, 사용자가 **수정한 최종 결과**와의 매핑이 부족하여 "오인식 사례"를 자동으로 추출하기 어려움.
- **도메인 특화 처리 부재**: 일반적인 STT는 "삼천원"을 잘 인식하지만, "3,000원" 포맷팅이나 "스벅(스타벅스)" 같은 약어 처리는 별도 후처리가 필요함.
- **방어 로직 미비**: 소음이 심하거나 무음인 경우에도 불필요하게 Whisper를 호출하거나 잘못된 결과를 노출함.
- **사용자 피드백 루프 부재**: 정오답 노트(데이터셋)를 만드는 과정이 수동적임.

## 3) 개선 전략 (단계별)

### A. 진단/로깅 체계 강화 (즉시 적용)
- **Session ID 기반 추적**: `Audio Record Start` -> `STT Result` -> `User Edit` -> `Final Save` 까지 하나의 세션으로 묶어 로그 기록.
- **수정 거리(Levenshtein Distance) 측정**: STT 결과와 사용자 최종 입력값 간의 차이를 계산하여, 차이가 큰 케이스를 "학습 대상"으로 자동 분류.
- **메타데이터 강화**: 녹음 당시의 `기기 볼륨`, `마이크 권한 상태`, `평균 진폭(Amplitude)` 등을 함께 기록.

### B. 품질 판별 및 Fallback 정교화 (단기)
- **Rule-based Validation**:
  - 텍스트 길이는 충분한데 **숫자(금액)**가 없는 경우 -> Fallback (지출 기록 앱 특성상 금액 누락은 99% 실패)
  - "알 수 없음" 혹은 "의미 없는 문장(ㅋㅋㅋ 등)" 감지.
- **Confidence Scoring**:
  - Local STT 신뢰도 점수 + **Key Entity(금액, 날짜) 존재 여부**를 결합하여 최종 신뢰도 산출.

### C. 도메인 특화 텍스트 정규화 (Post-Processing)
- **발음 보정 딕셔너리**:
  - 자주 틀리는 브랜드명/지명 매핑 (예: "매가 커피" -> "메가커피", "편의 점" -> "편의점")
- **숫자/단위 정규화**:
  - 한글 숫자 -> 아라비아 숫자 변환 (이삼천원 -> 23,000원)
  - 날짜 상대 표현 해석 (오늘, 어제, 그저께 -> `YYYY-MM-DD` 로 변환하여 AI 프롬프트에 힌트로 제공)

### D. 데이터 수집: User-in-the-loop Flywheel (중기)
- **Passive Collection**: 사용자가 직접 수정한 데이터를 "정답 데이터(Ground Truth)"로 활용 (가장 고품질/저비용).
- **Reject Queue**: 신뢰도가 낮아 Whisper로 넘어갔던 오디오들을 별도로 모아, 추후 Local STT 튜닝셋으로 활용.

## 4) 데이터 수집 규모 권장 (단계별)
기존의 대규모 인력 투입(500명) 방식보다, **실사용 데이터 기반의 점진적 확장**을 권장함.

- **Phase 1 (Bootstrapping, 사내/지인)**
  - 목표: 10명, 인당 50문장 (총 500문장)
  - 용도: 초기 발음 보정 딕셔너리 구축 및 치명적 오류 수정
  - 방법: 개발팀 및 지인 대상 알파 테스트

- **Phase 2 (Flywheel, 실제 유저)**
  - 목표: 일일 수정 로그 100건 이상 자동 수집
  - 용도: 주기적인 재학습 및 엣지 케이스 발견
  - 방법: 약관 동의 하에 "오인식된 오디오 및 수정본" 익명 전송

## 5) 라벨링 규칙 (Entity Tagging)
- **Golden Rule**: 사용자의 **최종 의도**를 정답으로 간주한다.
- **Entity Types**:
  - `@PRICE`: 금액 (화폐 단위 포함/미포함)
  - `@DATE`: 날짜 (절대/상대)
  - `@PLACE`: 상호명, 장소
  - `@CATEGORY`: 카테고리 (식비, 교통 등)
  - `@CONTENT`: 지출 내역 메모

## 6) Whisper Fallback 조건 (Updated)
비용/속도 최적화를 위해 아래 조건 중 하나라도 만족하면 Whisper 호출:
1. **Local STT 신뢰도 < 0.6** (임계값 튜닝 필요)
2. **필수 엔티티(@PRICE) 누락**: 문장 길이가 5어절 이상인데 숫자가 전혀 없는 경우.
3. **오디오 신호 이상**: SNR(신호 대 잡음비)은 양호하나 STT 결과가 공백인 경우 (모델 오류 의심).

## 7) 실행 계획 (Revised Schedule)
- **Week 1: 기반 마련 및 룰 베이스 보정**
  - 로깅 시스템 개선 (Session ID, User Edit 추적)
  - 정규화/발음 보정 딕셔너리 적용 (단순 치환으로 즉각적 효과)
  - Fallback 로직 개선 (금액 누락 시 재검사 등)
- **Week 2: 데이터 파이프라인 및 정밀 튜닝**
  - Phase 1 데이터 수집(사내) 및 검증
  - 수집된 오답 데이터를 기반으로 프롬프트(System Prompt) 튜닝
  - (옵션) 수집된 데이터로 Whisper 파인튜닝 가능성 타진

## 8) 검증 지표 (KPI)
- **Entity Recognition Rate (ERR)**: 금액/날짜/상호명을 정확히 인식했는가? (가장 중요)
- **Edit Distance Ratio**: (STT 결과) vs (유저 최종 저장 값) 간의 차이 감소율.
- **Whisper Call Ratio**: 전체 음성 시도 중 Whisper API 호출 비율 (목표: < 10% 유지).
